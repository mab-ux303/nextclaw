# NCP 定位与愿景

## 一句话定义

NCP 不是某一个 Agent 产品，而是一套用于构建 Agent 应用的通用 building blocks。

它的目标不是把某个 Demo 做大，而是把协议、传输、运行时、状态、框架绑定、UI 积木这些共性能力沉淀成可组合、可替换、可复用的基础设施。

## 定位

NCP 在整个项目中的定位是：

- Agent 应用层之下的通用构建层
- NextClaw 自身可以复用的一组底层积木
- 面向未来不同 Agent 场景的统一协议与装配基座

这意味着我们做 NCP，不是为了服务某一个页面、某一个 demo、某一种 chat 形态，而是为了让开发者能更低成本地构建不同形态的 Agent 应用，例如：

- Chat Agent
- Workflow Agent
- Copilot
- Multi-Agent 协作系统
- 嵌入业务系统的 Agent 面板

## 愿景

我们的愿景是：

让构建 Agent 应用这件事，从“每次重新造一套系统”，变成“像搭积木一样组装通用能力”。

最终理想状态是，开发者在构建 Agent 应用时可以清晰分层：

1. 用 `@nextclaw/ncp` 定义协议契约。
2. 用 transport/client/server 积木接入运行环境。
3. 用 toolkit/runtime/state 积木完成消息流与会话编排。
4. 用 `ncp-react` / `ncp-react-ui` 这类绑定层和展示层积木接入框架。
5. 在 `apps/*` 中只保留业务差异、品牌表达和场景特有逻辑。

如果这个愿景成立，那么一个新 Agent 应用的搭建过程应该更像“组装”，而不是“重写”。

## 我们到底在解决什么问题

在没有 NCP 之前，Agent 应用通常会反复出现几类问题：

- 每个应用都重复实现一套消息协议、流式事件和会话状态
- transport、runtime、UI、业务逻辑容易耦在一起，后续很难替换
- 某个 demo 里的实现一旦长大，就会被误当成通用层继续复用
- 想支持新框架、新传输协议或新交互形态时，需要大面积重写

NCP 的核心价值，就是把这些“本该通用、却总在重复”的部分提炼出来。

## 产品与架构判断

围绕 NCP，我们要坚持以下判断：

- 先做 building blocks，再做壳层应用
- 先保证分层清晰，再追求功能堆叠
- 先抽象跨场景共性，再接受业务差异留在应用层
- NextClaw 是 NCP 的消费者之一，但不应该反向绑死 NCP 的边界
- 产品体验上默认追求最佳易用性，尽量不要把配置文件、底层结构或内部实现细节直接暴露给用户；能通过产品化入口解决的，不应要求用户手改配置

## 推进原则

NCP 的发展推进，默认长期遵循以下原则：

- 永远优先面向通用场景，而不是围绕某个单点业务做特化抽象
- 永远优先可扩展、可维护、可组合的结构，而不是短期堆功能
- 永远优先业务解耦，让通用层与具体产品流程保持清晰边界
- 永远保持实现干净、简单，避免为了“看起来更完整”提前引入复杂层级

这几条不是阶段性建议，而是后续所有 NCP 设计和实现的长期约束。

如果某个方案虽然能更快落地，但会让 NCP 变得更业务化、更耦合、更难替换或更难维护，那么默认不应采用。

所以，NCP 的成功标准不是“某个 demo 看起来更完整”，而是：

- 新应用接入时是否更快
- 传输层是否可替换
- 框架绑定是否足够薄
- UI 积木是否能复用但不携带业务判断
- 应用层是否还能保持自由组装

## 与 NextClaw 的关系

NextClaw 可以看作 NCP 理念的一号真实消费方与验证场。

二者关系应该是：

- NCP 提供通用层
- NextClaw 负责产品化组装
- demo / product / internal tools 都通过消费 NCP 来验证抽象是否成立

这也意味着，当某段能力只服务某个产品流程时，它应该留在应用层；当某段能力已经具备跨场景复用价值时，才应该进入 NCP。

## 第一阶段战略目标

第一阶段的关键目标之一，是把 NCP 体系推进到足够完善的程度，使它可以承接并逐步替换 NextClaw 当前的 Agent 前后端体系。

更具体地说：

- 我们希望未来 NextClaw 的 agent frontend / backend 主体能力，能够建立在 NCP 之上
- NCP 需要具备足够完整的协议、传输、runtime、state、framework binding、UI building blocks，支撑这次迁移
- 这次迁移的目标不是机械搬运，而是用 NCP 的方式重构和替换原有实现
- 底层存储层需要保持一致性；在第一阶段，它可以作为相对独立的一层保留，不强制迁入 NCP

因此，第一阶段不是“做一个能跑的 demo”就结束，而是要用“是否足以支撑 NextClaw agent 体系迁移”来检验 NCP 的成熟度。

如果某项能力无法支撑这条迁移路径，或者只能通过引入大量 NextClaw 私有耦合才能成立，那么说明这项抽象还不够成熟。

## 非目标

为了避免边界漂移，NCP 默认不做这些事：

- 不把 demo 业务流程直接沉到通用包
- 不把产品级 presenter/store/navigation 当成基础设施输出
- 不为了短期兼容长期保留双实现
- 不提前抽象“未来也许会用到”的复杂层级

## 对后续设计的约束

后续所有 NCP 相关设计，都应该回到这份定位文档来判断：

1. 这是不是构建 Agent 应用的共性能力？
2. 这项能力应当落在协议、传输、toolkit、框架绑定、展示层还是应用层？
3. 这个抽象是否真的减少了调用方复杂度？
4. 这个设计是在沉淀 building blocks，还是在把某个 demo 私有实现包装成“通用”？

只要第 1、2、3、4 任一答案不成立，就不应该进入 NCP 通用层。
